# マラソンポータルサイト 要件定義書

## 1. プロジェクト概要

### 1.1 ビジョン
全国のマラソン大会情報を網羅し、ランナーが「大会を探す → 準備する → 遠征を楽しむ」までをワンストップで支援するWebサービス。多言語対応（日本語・英語）により、国内ランナーだけでなくインバウンドランナーの日本マラソン遠征もサポートする。

### 1.2 ターゲットユーザー
- **国内ランナー**：初心者〜サブ3狙いのシリアスランナー
- **インバウンドランナー**：日本のマラソン大会に参加したい海外ランナー
- **マラソン観光層**：大会参加＋観光・温泉を楽しみたいランナー

### 1.3 初期リリース方針
- プラットフォーム：**Webのみ（レスポンシブ対応）**
- 対応言語：**日本語 / 英語**
- モバイルアプリ（Flutter）は将来フェーズで検討

## 2. 機能要件

### 2.1 大会情報（コア機能）

| 項目 | 内容 | 優先度 |
|------|------|--------|
| 大会名 | 日本語・英語 | 必須 |
| 開催日 | 年月日 | 必須 |
| 開催地 | 都道府県・市区町村 | 必須 |
| 距離カテゴリ | フル / ハーフ / 10K / 5K / ウルトラ 等 | 必須 |
| 制限時間 | 距離カテゴリごとの制限時間 | 必須 |
| 参加賞 | テキスト＋写真＋カテゴリ分類（後述 2.8） | 必須 |
| エントリー情報 | 参加費、定員、エントリー開始/終了日 | 必須 |
| コースプロフィール | インタラクティブ地図＋高低差チャート（後述 2.9） | 必須 |
| エイドステーション | 設置距離、提供内容（特徴的なご当地エイド含む） | 必須 |
| 公式サイトURL | 外部リンク | 必須 |
| 受付情報 | 前日受付/当日受付、受付時間、受付場所 | 必須 |
| スタート時間 | 各距離カテゴリごと | 必須 |

### 2.2 アクセス情報

| 項目 | 内容 | 優先度 |
|------|------|--------|
| 最寄り駅 | 会場の最寄り駅（複数対応） | 必須 |
| 駅→会場アクセス | 徒歩/バス/シャトルバス等の手段と所要時間 | 必須 |
| 当日到着可否判定 | ユーザーの最寄り駅 → 会場最寄り駅への経路検索（後述） | 必須 |

### 2.3 当日到着可否判定機能（差別化機能）

ユーザーが自分の最寄り駅を設定することで、大会当日の受付時間またはスタート時間に間に合う公共交通手段が存在するかを判定する機能。

**入力**
- ユーザーの最寄り駅（事前設定 or 都度入力）
- 対象大会のスタート時間 / 受付開始時間

**出力**
- 間に合う/間に合わない（前泊推奨）の判定
- 最適な経路の概要（出発時刻、乗換回数、所要時間）
- 始発利用時の到着時刻

**技術要件**
- 乗換案内APIの利用が必要（候補は後述の技術選定セクションで整理）
- 終電/始発を考慮した逆引き検索
- オフライン時のフォールバック（「最寄り駅からの所要時間の目安」を静的データとして保持）

### 2.4 周辺情報

| 項目 | 内容 | 優先度 |
|------|------|--------|
| 近隣観光地 | 名称、概要、会場からの距離 | 必須 |
| 温泉・銭湯 | レース後のリカバリー向け施設情報 | 高 |
| ご当地グルメ | エリアの名物料理・店舗 | 高 |
| 宿泊エリア | 推奨宿泊エリアの概要 | 中 |

### 2.5 気象データ

| 項目 | 内容 | 優先度 |
|------|------|--------|
| 過去の気象データ | 開催月の平均気温・最高/最低気温・湿度・降水確率 | 高 |
| データソース | 気象庁の過去データ（CC BY 4.0）、Open-Meteo API（無料） | — |
| 表示形式 | 過去数年分のグラフ表示 | 高 |

### 2.6 大会検索・フィルタリング

| フィルタ項目 | 内容 |
|-------------|------|
| 開催月 | カレンダー形式 + 月別フィルタ |
| エリア | 地方 → 都道府県の階層フィルタ |
| 距離カテゴリ | フル / ハーフ / 10K / 5K / ウルトラ |
| タグ | 初心者おすすめ / 記録狙い（フラット）/ 景色が良い / ご当地エイド充実 等 |
| 制限時間 | 最低制限時間での絞り込み（例：6時間以上） |
| 当日アクセス | 自分の最寄り駅から当日到達可能な大会のみ表示 |
| エントリー状況 | エントリー受付中 / 近日開始 |
| 参加賞カテゴリ | メダル / タオル / Tシャツ / 物産品 等での絞り込み |

### 2.7 多言語対応

| 要件 | 詳細 |
|------|------|
| 対応言語 | 日本語（デフォルト）/ 英語 |
| 翻訳範囲 | UI文言、大会名、大会概要、周辺情報 |
| URL設計 | `/ja/races/...` `/en/races/...` または サブドメイン |
| 実装方式 | i18nライブラリ（next-intl, react-i18next 等） |
| コンテンツ翻訳 | 大会データの英語フィールドをJSONに保持 |

### 2.8 参加賞カテゴリ機能

大会の参加賞をカテゴリ分類し、「メダルがもらえる大会」「ご当地物産品がもらえる大会」など参加賞を軸にした大会検索を可能にする。

**参加賞カテゴリ（初期定義）**

| カテゴリID | カテゴリ名（日本語） | カテゴリ名（英語） | 例 |
|-----------|-------------------|-------------------|-----|
| medal | メダル / 完走メダル | Finisher Medal | 完走メダル、記念メダル |
| tshirt | Tシャツ | T-Shirt | 大会記念Tシャツ、ドライTシャツ |
| towel | タオル | Towel | フィニッシャータオル、スポーツタオル |
| local_product | 物産品 / ご当地品 | Local Product | 地元の特産品、名産品詰め合わせ |
| food | 食品 | Food / Snack | お米、お菓子、地元食材 |
| goods | グッズ / 記念品 | Goods | ポーチ、キャップ、ボトル |
| coupon | クーポン / 割引券 | Coupon / Voucher | 温泉割引券、地域商品券 |
| other | その他 | Other | 上記に該当しないもの |

**仕様**
- 1つの大会に対して複数カテゴリを設定可能（例：メダル＋Tシャツ＋物産品）
- 検索フィルタで複数カテゴリのAND/OR検索に対応
- カテゴリは将来的に拡張可能な設計とする

### 2.9 コースプロフィール機能

大会公式サイトのコース説明（通過ポイント・道順）を元に、独自にルートデータを作成し、インタラクティブな地図表示と高低差チャートを提供する。公式の画像を転載するのではなく、公開されている事実情報（コースが通る道路）を元に独自にルートを作成するため、著作権上の問題を回避できる。

**表示コンポーネント**
- インタラクティブ地図：ルートをポリラインで描画（Leaflet + OpenStreetMap）
- 高低差チャート：横軸=距離(km)、縦軸=標高(m)のプロフィールチャート（Recharts）
- インタラクション連携：チャート上のマウスホバーで地図上の該当地点をハイライト
- エイドステーション・距離表示のオーバーレイ

**データ生成パイプライン**

```
[ルート作成（手動）]
  公式サイトのコース説明を参照し、地図ツール上でルートをトレース
  ツール候補：Googleマイマップ / Ride with GPS / JOSM
      ↓
  GPXファイルとしてエクスポート
      ↓
[標高データ付与（自動）]
  GPXの各ポイント（緯度経度）に対して標高APIから標高を取得
  API候補：
    - 国土地理院 標高タイルAPI（無料、高精度、日本全国カバー）
    - Open-Meteo Elevation API（無料、グローバル対応）
      ↓
[ビルド時変換（自動）]
  GPX + 標高データ → 静的JSON に変換
  JSONにはポイント配列（緯度, 経度, 標高, 累積距離）を格納
      ↓
[フロントエンド描画]
  JSON → Leaflet（地図）+ Recharts（高低差チャート）で描画
```

**ルート作成の作業量目安**
- 1大会あたり15〜30分（道路スナップ機能を活用）
- 公式コース説明の通過ポイントをクリックしていくだけで概ね正確なルートが描ける

**データ格納形式（JSON）**

```json
{
  "race_id": "tokyo-marathon-2026",
  "distance_km": 42.195,
  "total_elevation_gain_m": 45,
  "total_elevation_loss_m": 39,
  "points": [
    { "lat": 35.6895, "lng": 139.6917, "ele": 8.2, "dist_km": 0.0 },
    { "lat": 35.6890, "lng": 139.6920, "ele": 8.5, "dist_km": 0.05 }
  ]
}
```

## 3. 決定事項

### 3.1 データ管理方式：JSONファイル + SSG

大会データはJSONファイルとしてリポジトリ内で管理し、Next.jsのSSG（静的サイト生成）でビルド時にHTMLを生成する。

**方式の詳細**
- 大会データ、エイド情報、アクセス情報、周辺スポット、気象データ等を構造化JSONファイルとして管理
- GPXファイル（コースデータ）もリポジトリに格納
- ビルド時にJSON + GPX → 静的HTML + 静的JSONを生成
- データ更新時はJSONファイルを編集してデプロイ

**パフォーマンス評価**
- 全国の主要マラソン大会は年間200〜300大会程度
- 1大会あたり約5KBとして、全体で1〜2MB程度
- SSGにより、リクエスト時のデータ読み込みは発生しない
- クライアントサイドのフィルタリングも300件程度なら瞬時に処理可能

**メリット**
- 外部サービスへの依存ゼロ（CMS障害やAPI制限の影響を受けない）
- Gitでデータのバージョン管理が可能（変更履歴の追跡、ロールバック）
- ローカル開発・テストが容易
- ランニングコストが最小限（ホスティング費用のみ）

**将来の移行パス**
- 大会数が1,000件を超える、または複数人での同時編集が頻発する場合はヘッドレスCMS（microCMS等）へ移行
- Phase 2でユーザーデータ（レビュー、認証情報）が必要になった段階でSupabase等のDBを追加
- JSONからの移行は構造が明確なため比較的容易

### 3.2 ユーザー認証

現時点では未決定。以下の機能との依存関係を考慮して判断が必要。

| 認証が必要な機能 | 認証なしの代替手段 |
|-----------------|------------------|
| 最寄り駅の保存 | ブラウザのlocalStorageに保存（端末依存） |
| 大会レビュー投稿 | 認証必須（スパム対策の観点） |
| お気に入り大会の保存 | localStorageで代替可能 |
| 参加予定大会の管理 | localStorageで代替可能 |

**推奨方針**：初期リリースはlocalStorageベースで最寄り駅・お気に入りを実装し、レビュー機能はPhase 2でユーザー認証と同時に導入する。認証基盤はSNSログイン（Google / Apple）を想定。

## 4. データモデル（概念設計）

### 4.1 大会（Race）

```
Race
├── id: string (PK, 例: "tokyo-marathon-2026")
├── name_ja: string
├── name_en: string
├── date: date
├── prefecture: string (都道府県コード)
├── city_ja: string
├── city_en: string
├── description_ja: text
├── description_en: text
├── official_url: string
├── entry_fee: number | null (全カテゴリ共通の場合に設定。カテゴリ別料金の場合はnull)
├── entry_fee_by_category: boolean (true=カテゴリごとに参加費が異なる)
├── entry_capacity: number (全カテゴリ合計の定員)
├── entry_start_date: date
├── entry_end_date: date
├── reception_type: enum (pre_day / race_day / both / pre_mail / none)
│   ※ pre_day=前日受付のみ, race_day=当日受付のみ, both=前日・当日両方,
│     pre_mail=事前郵送（前日受付あり）, none=事前郵送のみ（受付なし）
├── reception_note_ja: text (受付に関する補足説明)
├── reception_note_en: text
├── tags: string[] (初心者向け, フラット, 景色良い, ウルトラマラソン, etc.)
├── course_gpx_file: string | null (GPXファイルパス。未作成の場合はnull)
├── course_info: CourseInfo (コースの概要情報)
├── categories: RaceCategory[]
├── aid_stations: AidStation[]
├── checkpoints: Checkpoint[] (関門情報)
├── access_points: AccessPoint[]
├── nearby_spots: NearbySpot[]
├── weather_history: WeatherHistory[]
├── participation_gifts: ParticipationGift[]
├── created_at: datetime
└── updated_at: datetime
```

### 4.2 距離カテゴリ（RaceCategory）

```
RaceCategory
├── distance_type: enum (full / half / 10k / 5k / ultra / other)
├── distance_km: number
├── time_limit_minutes: number
├── start_time: time (最初のウェーブのスタート時刻)
├── capacity: number
├── entry_fee: number | null (カテゴリ別料金の場合に設定)
├── entry_fee_u25: number | null (U25等の割引料金がある場合)
├── name_ja: string | null (カテゴリ固有名称。例: "FUJI 5LAKES 120km")
├── name_en: string | null
├── description_ja: text | null (カテゴリの説明・コース概要)
├── description_en: text | null
└── waves: Wave[] | null (ウェーブスタートがある場合)

Wave (同一カテゴリ内の時間差スタート)
├── wave: string (ウェーブ識別子。例: "A", "B", "WAVE1")
├── start_time: time
└── end_time: time (制限時刻)
```

**設計判断の背景**：サンプルデータ作成時に、ウルトラマラソン（チャレンジ富士五湖）では同一カテゴリ内でも複数のウェーブスタートがあり、それぞれ制限時間の起算が異なることが判明した。また、カテゴリごとに参加費が異なるケースや、U25割引などの料金体系も存在するため、これらを `RaceCategory` 内で管理する設計とした。

### 4.8 コース概要（CourseInfo）

```
CourseInfo (大会データ内に埋め込み)
├── max_elevation_m: number (最高標高)
├── min_elevation_m: number (最低標高)
├── elevation_diff_m: number (高低差)
├── surface: enum (road / trail / mixed)
├── certification: string[] (例: ["JAAF", "AIMS"])
├── highlights_ja: text (コース上の見どころ)
├── highlights_en: text
├── notes_ja: text | null (携行品必須等の重要注意事項)
└── notes_en: text | null
```

**設計判断の背景**：コースの高低差や路面種別は大会選びの重要な判断材料。特にウルトラマラソンでは携行品必須（ボトル・カップ・補給食・防寒具等）などの重要な注意事項があり、`notes` フィールドで管理する。

### 4.9 関門情報（Checkpoint）

```
Checkpoint
├── distance_km: number (スタートからの距離)

```

### 4.10 コースプロフィール（CourseProfile）

```
CourseProfile (ビルド時にGPX + 標高APIから自動生成)

├── distance_km: number
```

### 4.11 ユーザー設定（localStorage）

```

├── preferred_language: enum (ja / en)

```

## 5. データ取得方針

### 5.1 データ種別ごとのリスク・取得方法一覧

| データ項目 | 著作権リスク | 取得方法 | 備考 |
|-----------|------------|---------|------|
| 大会基本情報（開催日・距離・制限時間・参加費・定員） | 低 | 公式サイトから手動で転記 | 事実情報の転記は問題なし |
| スタート時間・受付情報 | 低 | 公式サイト / 募集要項から手動転記 | 同上 |
| 参加賞情報 | 低 | 公式サイト / 募集要項から手動転記 | 事実の記載。写真は自前撮影 or フリー素材 |
| エントリー情報 | 低 | 公式サイトから手動転記 | 期日・金額等の事実情報 |
| コースプロフィール | **要注意** | 公式コース説明を参照し独自にGPXを作成（後述5.2） | 公式画像の転載はNG。独自作成で回避 |
| エイドステーション | 中 | 公式サイトの情報を参考に独自の文章で記載 | 表現のコピーはNG。事実を自分の言葉で |
| 最寄り駅・アクセス | 低 | 公式サイト + 地図情報から整理 | 事実情報 |
| 周辺スポット（観光・温泉・グルメ） | **要注意** | 独自に説明文を作成 | 観光協会サイト等からの転載はNG |
| 気象データ | 低 | 気象庁 過去データ（CC BY 4.0）/ Open-Meteo API | 出典明記で自由利用可 |
| 大会写真・画像 | **要注意** | 自前撮影 / フリー素材 / UGC（Phase 2） | 他者の写真は無断利用NG |

### 5.2 コースプロフィール生成パイプライン

公式のコースマップや高低差図は著作物として保護されている可能性が高く、転載はできない。代わりに、公式サイトで公開されているコース説明（通過ポイント・道順）を参考に、地図上で独自にルートをトレースしてGPXファイルを作成する。道路のルート自体は事実であり著作物ではないため、このアプローチは著作権上問題がない。

**Step 1：ルート作成（手動・1大会15〜30分）**
- 公式サイトのコース説明（「○○交差点を右折」「○○川沿いを北上」等）を参照
- 地図ツール上で通過ポイントをクリックしてルートをトレース
- 道路スナップ機能により、交差点をクリックするだけで正確なルートが描ける
- ツール候補：Googleマイマップ、Ride with GPS、JOSM（OpenStreetMap エディタ）
- GPXファイルとしてエクスポートし、`/data/gpx/` に格納

**Step 2：標高データ付与（自動スクリプト）**
- GPXの各ポイント（緯度経度）に対して標高APIから標高を取得
- API候補：
  - **国土地理院 標高タイルAPI**（推奨）：無料、高精度（5mメッシュ）、日本全国カバー
  - **Open-Meteo Elevation API**：無料、グローバル対応（海外大会にも対応可能）
- ビルドスクリプト（Node.js）で自動処理

**Step 3：静的JSON変換（ビルド時自動）**
- GPX + 標高データ → CourseProfile JSON に変換
- 累積距離を計算し、各ポイントに `dist_km` を付与
- 合計獲得標高（elevation gain/loss）を自動算出
- 出力先：`/data/course-profiles/{race-id}.json`

**Step 4：フロントエンド描画**
- Leaflet + OpenStreetMap：ルートをポリラインで地図上に描画
- Recharts：高低差プロフィールチャート（横軸=距離、縦軸=標高）
- インタラクション連携：チャートのマウスホバーで地図上の対応地点をハイライト
- エイドステーションの位置をチャート・地図の両方にマーカー表示

### 5.3 気象データ取得

- **データソース**：気象庁の過去気象データ（CC BY 4.0ライセンス、出典明記で自由利用可）
- **補助ソース**：Open-Meteo Historical Weather API（無料、REST API）
- **取得方法**：大会開催地の最寄りアメダス観測所の、開催月における過去5〜10年分のデータを取得
- **更新頻度**：年1回（前年のデータを追加）
- **注意点**：出典として「気象庁」または「Open-Meteo」を必ず明記

### 5.4 周辺スポット情報

- 観光地、温泉、グルメ等の説明文は**すべて独自作成**（他サイトからの転載はNG）
- 施設名・住所・営業時間等の事実情報は記載可能
- 将来的にGoogle Places APIからの基本情報取得も検討（API利用規約に準拠）
- 写真は自前撮影、フリー素材、または施設の許諾を得て掲載

### 5.5 初期データ投入戦略

| フェーズ | 大会数 | 対象 | 目的 |
|---------|-------|------|------|
| α版 | 10〜20大会 | 主要都市マラソン（東京、大阪、名古屋、福岡等）＋特徴的な地方大会 | サイトの価値実証、UI/UXの検証 |
| β版 | 50〜100大会 | 全国の主要大会を網羅 | 本格運用開始、SEO効果の確認 |
| 正式版 | 200〜300大会 | 全国の公認・準公認大会を広くカバー | フル稼働 |

**データ投入の作業量見積もり**
- 基本情報の入力：1大会あたり30〜60分
- GPXルート作成：1大会あたり15〜30分
- 周辺スポット調査・執筆：1大会あたり30〜60分
- 合計：1大会あたり約1.5〜2.5時間
- α版（20大会）：約30〜50時間の作業

### 5.6 データ収集における禁止事項

- RUNNET等の大会情報サイトからのスクレイピング（利用規約違反）
- 公式サイトのコースマップ画像・高低差図の転載（著作権侵害の可能性）
- 観光協会・食べログ等からの説明文の転載（著作権侵害）
- 他者が撮影した写真の無断使用

### 5.7 将来的なデータ収集の効率化

- **大会主催者向け登録フォーム**：サイトの認知度が上がった段階で、主催者が自らデータを登録・更新できる仕組みを提供（Phase 3）
- **ランナーコミュニティとの連携**：GPXルートデータの共有や、周辺スポットのクラウドソーシング（Phase 2〜3）
- **AI活用**：公式サイトの情報を元に、基本データのドラフト生成を自動化（人間が最終確認・修正）

### 6.1 フロントエンド

| 要素 | 推奨技術 | 理由 |
|------|---------|------|
| フレームワーク | **Next.js (App Router)** | SSG対応、i18nルーティング、Reactエコシステム |

| 地図表示 | **Leaflet + OpenStreetMap** | 無料、コースマップ・周辺スポット表示 |
| グラフ | **Recharts** | 高低差チャート・気象データの可視化、Reactとの親和性 |

### 6.2 データ管理

| 要素 | 技術 | 理由 |
|------|------|------|
| 大会データ | **JSONファイル（リポジトリ内）** | 外部依存ゼロ、Git管理、ランニングコスト最小 |
| コースデータ | **GPXファイル → ビルド時にJSON変換** | 標準フォーマット、地図ツールとの互換性 |
| 標高データ | **国土地理院 標高タイルAPI** | 無料、高精度（5mメッシュ） |
| ホスティング | **Vercel** | Next.jsとの最適化、エッジ配信 |
| 画像 | リポジトリ内 or Cloudinary | 初期はリポジトリ内、画像数増加時にCDN移行 |

### 6.3 外部API連携

| API | 用途 | 候補 |
|-----|------|------|
| 経路検索 | 当日到着可否判定 | NAVITIME API / 駅すぱあと API / Google Directions API |
| 標高データ | コースプロフィール生成 | 国土地理院 標高タイルAPI / Open-Meteo Elevation API |
| 気象データ | 過去の気象データ取得 | 気象庁 CSV（CC BY 4.0）/ Open-Meteo Historical API |
| 地図タイル | コースマップ表示 | OpenStreetMap（無料） |

#### 経路検索API比較

| API | 日本の公共交通 | 始発/終電対応 | 英語対応 | 料金目安 |
|-----|--------------|-------------|---------|---------|
| NAVITIME API | ◎ | ◎ | ○ | 有料（従量課金） |
| 駅すぱあと API | ◎ | ◎ | △ | 有料（月額） |
| Google Directions (transit) | ○ | △ | ◎ | 有料（$5/1000リクエスト） |

**推奨**：日本の公共交通に特化した NAVITIME API が最も精度が高い。英語対応も考慮するとGoogle Directions APIとの併用も選択肢。

## 付録A. サンプルデータ作成からの設計フィードバック

サンプルデータとして長野マラソン（都市型フルマラソン）とチャレンジ富士五湖（ウルトラマラソン）の2大会分のJSONを作成した結果、以下のデータモデル改善を行った。

### A.1 参加費の柔軟な管理

| パターン | 例 | 対応 |
|---------|-----|------|
| 全カテゴリ共通料金 | 長野マラソン（14,300円） | `entry_fee` に金額を設定 |
| カテゴリ別料金 | 富士五湖（120km: 25,000円、100km: 23,000円 等） | `entry_fee: null`, `entry_fee_by_category: true`, 各 `RaceCategory` 内に `entry_fee` を設定 |
| U25等の割引料金 | 富士五湖U25（120km: 20,000円 等） | 各 `RaceCategory` 内に `entry_fee_u25` を設定 |

### A.2 受付方式の拡張

初期設計では「前日 / 当日 / 両方」の3択だったが、実データでは以下のバリエーションが必要だった。

| reception_type | 説明 | 例 |
|---------------|------|-----|
| pre_day | 前日受付のみ | — |
| race_day | 当日受付のみ | — |
| both | 前日・当日両方 | — |
| pre_mail | 事前郵送＋前日受付あり | チャレンジ富士五湖 |
| none | 事前郵送のみ（受付なし） | 長野マラソン（第27回〜） |

補足情報を `reception_note_ja/en` で管理する設計とした。

### A.3 ウェーブスタート対応

ウルトラマラソンでは同一距離カテゴリ内で複数のウェーブスタートがあり、各ウェーブで制限時刻が異なる。`Wave` モデルを `RaceCategory` 内にネストする形で対応した。

### A.4 コース概要情報（CourseInfo）の追加

GPXから自動生成する `CourseProfile` とは別に、手動入力の概要情報として `CourseInfo` を追加した。高低差、路面種別、公認情報、コースの見どころ、注意事項（携行品必須等）を管理する。

### A.5 関門情報（Checkpoint）の追加

制限時間が厳しい大会（長野マラソンの5時間等）では、関門の距離と閉鎖時刻がランナーの大会選びに直結する重要情報であるため、独立したモデルとして追加した。

### A.6 作成済みサンプルデータ

| ファイル | 大会 | 特徴 |
|---------|------|------|
| `data/races/nagano-marathon-2026.json` | 第28回 長野マラソン | 都市型フルマラソン、単一カテゴリ、関門9箇所、事前郵送（受付なし） |
| `data/races/challenge-fuji5lakes-2026.json` | 第36回 チャレンジ富士五湖 | ウルトラマラソン、4カテゴリ（62km〜120km）、ウェーブスタート、カテゴリ別料金・U25料金 |
